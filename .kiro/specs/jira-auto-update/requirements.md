# Requirements Document

## Introduction

This feature implements a commit-and-push-triggered Jira auto-update system using a combination of a Git commit-msg hook and Kiro Agent Hooks. When a developer commits code, the system validates the commit message contains a Jira ticket ID and posts an LLM-generated diff summary as a comment on the Jira ticket. When the developer pushes, the system transitions the ticket status to "In Review". This separation ensures that small/incremental commits build context on the ticket without prematurely changing its status, while the push event signals that work is ready for review.

## Glossary

- **Hook_System**: The combined system of a Git commit-msg hook (for validation), a post-commit Kiro Agent Hook (for Jira comments), and a post-push Kiro Agent Hook (for Jira status transition) that processes commits and pushes automatically.
- **Ticket_ID**: A Jira issue key matching the pattern of uppercase letters followed by a hyphen and digits (e.g., PROJ-482, ABC-123).
- **Commit_Message**: The full text of a Git commit message including subject line and body.
- **Diff_Summary**: A concise 2-4 sentence plain-English summary generated by an LLM describing what changed, why it changed, and whether tests were modified.
- **Jira_Client**: The component responsible for communicating with Jira via MCP server or REST API to transition ticket status and post comments.
- **LLM_Summarizer**: The component that accepts a commit diff and produces a human-readable summary of the changes.
- **Commit_Validator**: The commit-msg Git hook component that validates the presence of a Ticket_ID in the Commit_Message.

## Requirements

### Requirement 1: Commit Message Validation

**User Story:** As a developer, I want commits without a Jira ticket ID to be rejected at commit time, so that every committed change is traceable to a Jira ticket.

#### Acceptance Criteria

1. WHEN a developer runs git commit with no Ticket_ID anywhere in the Commit_Message, THE Commit_Validator SHALL reject the commit by exiting with a non-zero status code.
2. WHEN a developer runs git commit with no Ticket_ID in the Commit_Message, THE Commit_Validator SHALL print an error message to stderr explaining the required Ticket_ID format with an example such as "PROJ-123: Your commit description".
3. WHEN a developer runs git commit with a valid Ticket_ID in the Commit_Message, THE Commit_Validator SHALL allow the commit to proceed by exiting with a zero status code.
4. THE Commit_Validator SHALL identify a valid Ticket_ID using the regex pattern `/\b[A-Z][A-Z0-9]+-\d+\b/`.
5. WHEN a Commit_Message contains a valid Ticket_ID at any position including subject line or body, THE Commit_Validator SHALL accept the commit.
6. WHEN a merge commit is created via git merge, THE Commit_Validator SHALL apply the same Ticket_ID validation to the merge commit message.

### Requirement 2: Ticket ID Extraction

**User Story:** As a developer, I want the system to extract Jira ticket IDs from my commit message automatically, so that the correct ticket gets updated without additional manual input.

#### Acceptance Criteria

1. WHEN a Commit_Message contains one or more Ticket_IDs matching the pattern `/\b[A-Z][A-Z0-9]+-\d+\b/`, THE Hook_System SHALL extract a deduplicated list of unique Ticket_IDs regardless of how many times each ID appears in the message.
2. WHEN a Commit_Message contains multiple distinct Ticket_IDs (up to a maximum of 5), THE Hook_System SHALL update all matched tickets with the same Diff_Summary and status transition.
3. WHEN a Commit_Message contains multiple distinct Ticket_IDs, THE Hook_System SHALL log the list of all detected Ticket_IDs for developer visibility.
4. IF a Commit_Message contains more than 5 distinct Ticket_IDs, THEN THE Hook_System SHALL process only the first 5 Ticket_IDs in order of appearance and log a warning indicating that additional Ticket_IDs were ignored.

### Requirement 3: LLM Diff Summary Generation

**User Story:** As a code reviewer, I want an automatically generated summary of commit changes posted to the Jira ticket, so that I have immediate context about what changed without reading the full diff.

#### Acceptance Criteria

1. WHEN a commit containing a valid Ticket_ID succeeds, THE LLM_Summarizer SHALL generate a Diff_Summary from the commit diff, the Commit_Message, and the list of changed files.
2. THE LLM_Summarizer SHALL produce a Diff_Summary in plain text that is 3-4 sentences in length covering what changed, why it changed if the reason is explicitly stated in the Commit_Message or diff, and whether files matching the project's test directory or test naming convention were added or modified.
3. WHEN the commit diff exceeds the LLM context window size, THE LLM_Summarizer SHALL summarize individual files first and then produce a combined summary from those file summaries that is still 2-4 sentences in length.
4. THE LLM_Summarizer SHALL use the output of `git show <sha> --no-color` as the diff input for summary generation.
5. THE LLM_Summarizer SHALL NOT include information, assumptions, or reasoning that is not directly present in the commit diff or Commit_Message.

### Requirement 4: Jira Ticket Status Transition (on Push)

**User Story:** As a developer, I want my Jira ticket to automatically transition to "In Review" only when I push my branch, so that incremental commits don't prematurely change the ticket status.

#### Acceptance Criteria

1. WHEN a developer runs git push and the pushed commits contain valid Ticket_IDs, THE Jira_Client SHALL transition the corresponding Jira tickets' status to "In Review".
2. THE Jira_Client SHALL collect all unique Ticket_IDs from the commit messages in the pushed commits and transition each one.
3. THE Jira_Client SHALL look up available transitions on the ticket and match the target status by name using case-insensitive comparison rather than a hardcoded transition ID.
4. IF the target status "In Review" is not available as a valid transition for the ticket's current state, THEN THE Jira_Client SHALL log a warning indicating the transition is not available from the current status and skip the status change.
5. IF the ticket is already in "In Review" status, THEN THE Jira_Client SHALL skip the transition and not log a warning.
6. IF multiple transitions match "In Review" (case-insensitive), THEN THE Jira_Client SHALL use the first matching transition returned by the Jira API.
7. THE Hook_System SHALL NOT transition ticket status on commit — only on push.

### Requirement 5: Jira Comment Posting (on Commit)

**User Story:** As a code reviewer, I want an automated summary comment posted to the Jira ticket on each commit, so that I can track incremental progress without context-switching to the codebase.

#### Acceptance Criteria

1. WHEN a commit containing a valid Ticket_ID succeeds, THE Jira_Client SHALL post the Diff_Summary as a comment on the corresponding Jira ticket immediately after the commit (not waiting for push).
2. THE Jira_Client SHALL format the comment with the prefix `🤖 Auto-summary from commit <short-sha>:` followed by the Diff_Summary, where short-sha is the first 7 characters of the commit SHA.
3. THE Jira_Client SHALL format comments in plain text or Jira wiki markup format, not raw Markdown.
4. WHEN a Commit_Message contains multiple Ticket_IDs, THE Jira_Client SHALL post the same comment to each matched ticket independently.

### Requirement 6: Jira Authentication and Configuration

**User Story:** As a developer, I want Jira credentials stored securely outside version control, so that sensitive tokens are never accidentally committed to the repository.

#### Acceptance Criteria

1. THE Hook_System SHALL read Jira API credentials from the environment variables JIRA_API_TOKEN, JIRA_EMAIL, and JIRA_BASE_URL, or from a local `.env` secrets file containing the same keys.
2. WHEN both environment variables and a local secrets file provide credentials, THE Hook_System SHALL use environment variables as the authoritative source, ignoring the secrets file values.
3. IF the secrets file exists but is not listed in .gitignore, THEN THE Hook_System SHALL refuse to read credentials from the secrets file and SHALL log a warning indicating the file is not git-ignored.
4. IF none of the required credentials (JIRA_API_TOKEN, JIRA_EMAIL, JIRA_BASE_URL) are available from environment variables or the secrets file, THEN THE Hook_System SHALL log an error message indicating which credentials are missing and SHALL skip all Jira operations.
5. THE Hook_System SHALL attempt to connect to Jira via the Jira MCP server first for all Jira operations.
6. IF the Jira MCP server does not respond within 10 seconds or returns a connection error, THEN THE Hook_System SHALL fall back to the Jira REST API using HTTP Basic Auth with the configured API token and email.
7. THE Hook_System SHALL not log or echo Jira credentials or tokens in any output.

### Requirement 7: Error Handling for Jira Operations

**User Story:** As a developer, I want Jira update failures to be handled gracefully after a successful commit, so that my development workflow is never blocked by Jira API issues.

#### Acceptance Criteria

1. IF a matched Ticket_ID does not exist in Jira (404 response), THEN THE Jira_Client SHALL log a warning message including the Ticket_ID and allow the commit to remain intact without reverting.
2. IF the Jira API or MCP call fails due to a network error, timeout, or 5xx server error on the first attempt, THEN THE Jira_Client SHALL retry the operation once after a 2-second backoff delay.
3. IF the Jira API or MCP call fails after the retry attempt, THEN THE Jira_Client SHALL log a clear failure message and terminate gracefully with exit code 0 without crashing the hook.
4. IF the LLM call fails or does not respond within 30 seconds, THEN THE Hook_System SHALL fall back to posting a minimal comment listing only the changed files from the commit.
5. THE Hook_System SHALL not block or revert a completed commit due to any Jira or LLM operation failure.
6. IF the Jira API returns a 401 or 403 authentication error, THEN THE Jira_Client SHALL log an error indicating invalid credentials and SHALL NOT retry the operation.

### Requirement 8: Developer Logging and Visibility

**User Story:** As a developer, I want clear logging output after each commit, so that I can verify the auto-update succeeded or understand what went wrong.

#### Acceptance Criteria

1. WHEN the Hook_System completes all Jira operations successfully for a ticket, THE Hook_System SHALL log a success message to stdout that includes the Ticket_ID and confirms the status transition and comment were posted.
2. WHEN the Hook_System encounters a warning condition, THE Hook_System SHALL log a message to stderr that includes the Ticket_ID, the operation that triggered the warning, and the reason the operation was skipped or degraded.
3. WHEN the Hook_System encounters an error condition, THE Hook_System SHALL log a message to stderr that includes the Ticket_ID, the operation that failed, and the error reason returned by the failing component.
4. THE Hook_System SHALL prefix all log messages with the tag `[jira-hook]` to distinguish hook output from other Git output.
5. WHEN a Commit_Message contains multiple Ticket_IDs, THE Hook_System SHALL log a separate status message for each Ticket_ID indicating its individual operation outcome.
